原有架构

原有架构存在的问题

安全正确性不足

  • 早期更关注轨迹是否发出，而不是发出前是否全路径无碰撞。
  • 局部路径、FMP 后轨迹、控制器执行结果之间缺少统一门禁。

平台迁移复杂

  • MATLAB 中轨迹更接近任务空间演示；ROS2 中控制器接收关节轨迹。
  • 任务空间到关节空间的 IK、FK、时间尺度都需要重新对齐。

调参难度高

  • 障碍物尺寸、RRT 步长、边采样、via 插值密度相互耦合。
  • 一个参数变化可能同时影响绕障效果、平滑性和控制器可执行性。

因此本阶段工作的重点不是单纯让某一次动画看起来能绕开，而是把迁移后的系统变成可复核、可定位、可继续实验的工程系统。

当前最新架构

工作重点 1：C++ Runtime Bridge

为什么要做

  • Humble 环境中 moveit_py 可用性不稳定。
  • 真实机械臂仿真不能长期依赖简化 analytic 球体碰撞。
  • 碰撞检查和轨迹下发属于运行时关键路径，适合下沉到 C++ MoveIt2 接口。

具体改动

  • 新增 CheckStatesBatch 和 CheckMotionBatch 服务。
  • 新增 PlanLocalSegment 服务，提供 C++ RRT-Connect 局部规划。
  • 保留 Python fallback，方便开发阶段回退和对照。

工作重点 2：安全正确性补丁

修改点原因效果
FMP 后强制 CheckMotionBatch 避免调制后轨迹穿过障碍仍被下发 只有 states 和 edges 全部有效，才进入 dispatch
C++ RRT-Connect direct shortcut 无碰撞局部段不需要随机扰动 安全直连时返回 2 点路径，减少不必要 via
validatePath + 失败空路径 防止局部规划失败被当成成功路径 失败时 path_points=0，不进入 FMP
dispatch 安全门 控制器 ABORT 常来自起点断层和速度/加速度不合理 加入起点一致性、关节边界、速度/加速度估计与保守时间缩放

工作重点 3：最小评估体系

评价内容

  • 绕障成功性：invalid state / invalid edge count。
  • 轨迹平滑性：路径长度、最大关节跳变、jerk、速度、加速度。
  • 执行状态：action status、error code、是否 abort。
  • 局部路径复核：RRT stop reason、耗时、碰撞查询次数。

新增连通性增强

  • 评估前执行 collision backend preflight。
  • 区分“算法失败”和“后端不可用”。
  • 新增 result_interpretable 字段，避免误读 dry-run 结果。
  • 支持 planner output 到 evaluator 到图表的一条命令流程。

迁移难点：为什么这件事复杂

任务空间到关节空间

  • MATLAB 中更容易直接观察二维/三维轨迹。
  • ROS2 控制器需要关节轨迹，末端轨迹需要 IK 映射。
  • IK 解的连续性会影响后续 FMP 平滑性。

尺度映射

  • MATLAB 参数在抽象空间中较稳定。
  • ROS2 中障碍物尺寸、机械臂 link 尺寸、RRT 步长、edge resolution 都是物理尺度。
  • 这些尺度必须成组调整，不能孤立调参。

执行链路

  • 规划成功不等于控制器能跟踪。
  • Gazebo/RViz 可视化、MoveIt PlanningScene、控制器状态需要保持一致。
  • 因此我补了安全门和 evaluator，而不是只看单次动画。

论文阅读：pRRTC 是什么

论文核心

《pRRTC: GPU-Parallel RRT-Connect for Fast, Consistent, and Low-Cost Motion Planning》提出了一种面向 GPU 的并行 RRT-Connect。它不是简单多开几个 RRT，而是从树扩展、碰撞检测、最近邻搜索和内存访问方式上重新设计 RRT-Connect。

主要结果

  • 在 MotionBenchMaker 上最高约 10 倍加速。
  • 规划时间标准差最高降低约 5.4 倍。
  • 初始路径代价平均降低约 1.4 倍。

应用场景

  • 高自由度机械臂规划。
  • 狭窄通道和受约束任务。
  • 动态障碍物场景下实时 replanning。

pRRTC 的关键技术点

技术点论文做法对我项目的启发
双树并行扩展 多个 GPU block 同时执行 RRT-Connect 扩展，并使用 atomic 操作维护共享树。 说明局部规划瓶颈不只在采样策略，也在扩展和数据结构。
边碰撞检测并行 一条边上的多个离散配置同时检查，碰撞即早停。 对应我当前的 CheckMotionBatch states+edges，后续可以继续优化 edge validation。
最近邻并行搜索 线程分别搜索树节点子集，再 reduction 得到最近邻。 目前 C++ RRT-Connect 还偏基础，未来可考虑更高效最近邻或批量查询。
粗到细碰撞模型 使用球体包围模型进行低/高分辨率分层检测。 为后续 collision backend 性能优化和安全边界建模提供方向。

pRRTC 对本项目的影响

可以直接借鉴

  • RRT-Connect 比单向 RRT 更适合局部段快速连接。
  • edge validation 是关键，不应只检查离散节点。
  • 规划时间稳定性和路径质量都需要指标化，而不是只看是否成功。

暂不直接实现

  • GPU 并行不是当前阶段重点，当前先保证 ROS2 + MoveIt2 链路正确。
  • pRRTC 的球模型和 GPU 内存结构需要较大工程投入。
  • 我的近期目标是把 C++ runtime bridge 和 evaluator 做稳，为后续优化留接口。
